### Fase 2 — Schemas

### Fase 3 — Outputs oficiales
Objetivo:
hacer reales y coherentes los artefactos que ARV emite.

Outputs obligatorios:

- Certificate PDF
- Evidence Package ZIP

### Fase 4 — Corrección del demo
Objetivo:
alinear el demo con la base real del negocio.

Debe corregirse:
- naming vigente
- semántica local vs registered
- PDF funcional
- ZIP funcional
- QR funcional
- verify record
- ledger semantics
- textos visibles y branding

Resultado esperado:
el demo deja de prometer fuera de la base.

---

### Fase 5 — OSS oficial
Objetivo:
publicar el peón táctico.

Producto:
- ARV-Gig Evidence Locker

Debe incluir:
- captura/carga de evidencia
- hashing
- record local
- certificate básico
- package básico
- verificación local
- exportación

No debe incluir:
- autoridad soberana completa
- licenciamiento institucional
- ledger público oficial completo
- dispute package premium completo

Resultado esperado:
herramienta útil, adoptable y alineada con el negocio.
